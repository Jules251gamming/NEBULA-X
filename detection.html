01. Architecture de Compilation et Static Linking

Nebula-X est conçu via une chaîne de compilation Golang 1.21.5. À la différence des langages comme C# ou C++, Go utilise le static linking par défaut. Cela signifie que l'exécutable embarque son propre gestionnaire de mémoire (Garbage Collector) et sa propre pile réseau.

Analyse de la table des imports : L'outil ne fait pas appel aux API standards WININET.dll ou URLMON.dll. Il utilise syscall pour communiquer avec ntdll.dll. Cette discrétion est souvent mal interprétée par les moteurs de détection de type "Sandbox", qui ne voient pas d'activité DLL standard et classent le fichier comme "Anonyme/Suspect".

02. Gestion des Raw Sockets & Network Stack

Le noyau de Nebula-X repose sur l'injection de paquets via des sockets bruts (Raw Sockets). Cette technologie est indispensable pour tester les protections de couche 3 (Network Layer) et couche 4 (Transport Layer).

0441 func InitializeSocket() {
0442    fd, _ := syscall.Socket(syscall.AF_INET, syscall.SOCK_RAW, syscall.IPPROTO_RAW)
0443    // Configuration du Header IP_HDRINCL pour forgeage manuel
0444    syscall.SetsockoptInt(fd, syscall.IPPROTO_IP, syscall.IP_HDRINCL, 1)
0445 }

Vrai motif de détection : L'accès au flag IP_HDRINCL est strictement surveillé par Microsoft. Un binaire utilisateur demandant ce privilège sans certificat de signature EV (Extended Validation) est automatiquement flaggé par l'IA heuristique comme "HackTool" ou "Riskware".

03. Analyse Heuristique des Syscalls Windows

L'audit a révélé que les moteurs Bkav et Skyhigh se basent sur une analyse de "comportement prédictif". Voici les vecteurs d'appels qui provoquent les faux positifs :

SYSCALL MONITORING

  • - NtCreateThreadEx (Multithreading)
  • - NtAllocateVirtualMemory (Heap management)
  • - NtDeviceIoControlFile (Socket control)

Ces appels sont légitimes dans un contexte de stress-test pour simuler 10,000 connexions concurrentes, mais sont identiques aux techniques utilisées par les malwares de type "DDoS-Bot" pour optimiser leur consommation CPU.

05. Analyse de l'Entropie et de la Section de Données

L'entropie d'un fichier mesure son degré de "désordre". Un malware chiffré a une entropie proche de 8.0. Nebula-X affiche une entropie de 6.4.

Alerte de Faux Positif :
Le moteur Skyhigh marque le binaire comme VBObfus uniquement parce que les sections .text et .data sont fusionnées lors du build final pour optimiser la vitesse de lecture en RAM. Ce n'est pas de l'offuscation, mais de l'optimisation de bas niveau.

06. Protocole de Non-Malveillance & Preuves d'Intégrité

Nebula-X est audité pour garantir qu'aucune fonction malveillante n'est présente. Voici la check-list d'intégrité :

  • Persistance : Le code source ne contient aucun appel à RegistrySetKey ou AutoRun.
  • Exfiltration : Aucune fonction de lecture disque (ReadDir, ReadFile) n'est liée à un export réseau.
  • Privilèges : Le manifest asInvoker empêche le logiciel de demander des droits administrateur (UAC) par défaut.

07. Glossaire Technique & Références

- HEURISTIC ANALYSIS : Méthode par laquelle un antivirus analyse le code pour trouver des schémas suspects sans signature connue.
- GOROUTINE : Thread léger géré par le runtime Go permettant un parallélisme massif (indispensable pour le flood).
- DWARF SYMBOLS : Format de données de débogage standard. Sa suppression (flag -w) est la cause majeure des détections d'entropie.
